Fix rules:

1. Prefer prompt changes in `code-analysis\_tom_project_tracking\observability-ai-chat` when the failure is assistant reasoning, workflow ordering, response format, or Datadog-monitor guidance.
2. Change application code only when the failure is caused by code behavior, missing tool wiring, state-machine handling, browser UI behavior, or API behavior.
3. Keep changes surgical and directly tied to the failing source row. Do not rewrite unrelated prompts.
4. Do not add compatibility shims or fallback paths for old local states.
5. Preserve current successful behavior from other Observability AI Chat workflows.
6. If changing code, add or update targeted tests when a reasonable test surface exists.
7. If changing prompts only, include a concise rationale in the final report; do not invent tests that do not exist.

Validation after changes:

- For prompt-only changes, run the smallest available static check or no-op validation that exists for the prompt-loading surface if one exists. Otherwise report that no executable prompt-only test exists.
- For React changes, run targeted `npm test -- --watchAll=false --runInBand --runTestsByPath ...` and/or `npm run build` from `code-analysis\readmeapp`.
- For .NET backend changes, run the smallest relevant test project/test filter from `code-analysis` if available.
